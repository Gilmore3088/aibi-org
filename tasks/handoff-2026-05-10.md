## Outstanding items (priority order)

### 1. **BLOCKER** — Supabase Auth email templates hardcode `next=/dashboard`

**The bug:** When a user is sent to `/auth/login?next=/assessment/in-depth/take` (because they tried to access the In-Depth Assessment while logged out), `src/lib/supabase/auth.ts:83` correctly sets `emailRedirectTo` to include the requested `next` URL. But the magic-link email template ignores this — it hardcodes `next=/dashboard`. Same problem affects signup confirm, password reset, and email change.

**Result:** users who buy the In-Depth Assessment, click their magic-link email, get redirected to `/dashboard` instead of `/assessment/in-depth/take`. They have to manually re-navigate. Likely also breaks any future flow that wants to deep-link after auth.

**The fix:** in all 4 Supabase Auth email templates, change the `href=` from:
```
href="{{ .SiteURL }}/auth/callback?token_hash={{ .TokenHash }}&type=signup&next=/dashboard"
```
to:
```
href="{{ .RedirectTo }}&token_hash={{ .TokenHash }}&type=signup"
```
(replace `type=signup` per the template — `signup`, `magiclink`, `recovery`, `email_change`).

**Why this works:** `{{ .RedirectTo }}` renders the exact `emailRedirectTo` URL the API call passed in, which already includes the `next` query param. Appending `token_hash` + `type` to it preserves the requested redirect.

**How to apply:** the operator has to update the 4 templates in **Authentication → Email Templates** in the Supabase dashboard. There's no MCP for this — Supabase Auth config lives outside the MCP surface. Alternative path discussed (give the next agent a Supabase Personal Access Token; PATCH `/v1/projects/{ref}/config/auth` via Management API), but operator stopped before going that route.
