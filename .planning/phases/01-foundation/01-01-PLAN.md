**Part A: Append RLS policies to the migration file** (DB-06)

All policies MUST wrap `auth.uid()` in a SELECT subquery per CLAUDE.md performance pattern.

```sql
-- course_enrollments: users read/update their own enrollment
CREATE POLICY "Users read own enrollment" ON course_enrollments
  FOR SELECT TO authenticated
  USING ((select auth.uid()) = user_id);

CREATE POLICY "Users update own enrollment" ON course_enrollments
  FOR UPDATE TO authenticated
  USING ((select auth.uid()) = user_id)
  WITH CHECK ((select auth.uid()) = user_id);

-- activity_responses: users CRUD their own responses via enrollment ownership
CREATE POLICY "Users read own activity responses" ON activity_responses
  FOR SELECT TO authenticated
  USING (enrollment_id IN (
    SELECT id FROM course_enrollments WHERE user_id = (select auth.uid())
  ));

CREATE POLICY "Users insert own activity responses" ON activity_responses
  FOR INSERT TO authenticated
  WITH CHECK (enrollment_id IN (
    SELECT id FROM course_enrollments WHERE user_id = (select auth.uid())
  ));

-- work_submissions: users read/insert their own; reviewers read all
CREATE POLICY "Users read own submissions" ON work_submissions
  FOR SELECT TO authenticated
  USING (enrollment_id IN (
    SELECT id FROM course_enrollments WHERE user_id = (select auth.uid())
  ));

CREATE POLICY "Users insert own submissions" ON work_submissions
  FOR INSERT TO authenticated
  WITH CHECK (enrollment_id IN (
    SELECT id FROM course_enrollments WHERE user_id = (select auth.uid())
  ));

-- Note: Reviewer policies (read all submissions, update review fields) will be
-- added in Phase 7 when the reviewer identity model is decided (per STATE.md blocker).

-- certificates: public read for verification endpoint (no auth required)
CREATE POLICY "Public read certificates" ON certificates
  FOR SELECT TO anon, authenticated
  USING (true);

-- institution_enrollments: service role only (managed by API routes)
-- No user-facing policies needed — all access goes through server-side service role client.
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Supabase | Authenticated users write to their own rows via RLS |
| API routes -> Supabase | Service role client bypasses RLS for system writes |
| Public -> certificates | Unauthenticated reads of certificate verification data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | activity_responses | mitigate | RLS policy validates enrollment ownership via auth.uid() join — user cannot write to another user's enrollment |
| T-01-02 | Tampering | work_submissions | mitigate | review_scores and review_status columns have no user UPDATE policy — only service role can modify review fields |
| T-01-03 | Information Disclosure | certificates | accept | Certificate verification is intentionally public (name, designation, date, institution only — no PII beyond holder name) |
| T-01-04 | Elevation of Privilege | course_enrollments | mitigate | user_id column set by service role on enrollment creation — user UPDATE policy does not include user_id in WITH CHECK |
| T-01-05 | Denial of Service | institution_enrollments | mitigate | No user-facing RLS policies — all access via service role in API routes with request validation |
</threat_model>
