<objective>
Build the reviewer dashboard at /admin/reviewer/ with the 5-dimension rubric scoring interface and the review-submission API that enforces the Accuracy hard gate server-side. Failed submissions get written feedback and one resubmission; passing submissions transition to approved status (triggering certificate flow in Phase 8).

Purpose: This is the human quality gate that protects the AiBI-P credential. Without human review, the credential has no value. The Accuracy hard gate (score of 1 = auto-fail) ensures no fundamentally inaccurate work receives certification regardless of other dimension scores.

Output: Reviewer dashboard, rubric scoring UI, review-submission API, reviewer auth guard.
</objective>

Create `src/app/api/courses/review-submission/route.ts`:

**POST handler:**

1. **Auth**: Verify authenticated user via Supabase `getUser()`. Then call `verifyReviewer()` -- if not a reviewer, return 403.

2. **Parse + validate body**:
   - `submissionId`: required, non-empty string.
   - `scores`: required object with exactly 5 keys (`accuracy`, `completeness`, `tone_and_voice`, `judgment`, `skill_quality`), each an integer 1-4. Validate every field. Type-check against `ReviewScores`.
   - `feedback`: optional string (but see step 4 for conditional requirement).

3. **Load submission**: Use `createServiceRoleClient()` to fetch the submission by ID. Verify `review_status` is `'pending'` or `'resubmitted'` -- if already reviewed, return 409 "Submission already reviewed."

4. **Scoring logic (REVW-03, REVW-04)**:
   - Calculate `total = accuracy + completeness + tone_and_voice + judgment + skill_quality`.
   - **Accuracy hard gate (REVW-03)**: If `scores.accuracy === 1`, the result is ALWAYS `'failed'` regardless of total. This check happens FIRST, before total evaluation.
   - **Pass threshold (REVW-04)**: `total >= 14 AND scores.accuracy >= 3` = `'approved'`. Otherwise = `'failed'`.
   - If result is `'failed'` and `feedback` is missing or under 100 characters, return 400 "Written feedback required for failing submissions (minimum 100 characters)."

5. **Check resubmission eligibility (REVW-06)**: If result is `'failed'` and the submission's current `review_status` was `'resubmitted'` (meaning this is their second attempt), the learner gets NO further resubmissions. The feedback should note this is final. Set a `resubmission_exhausted` flag in the response.

6. **Write review**: Update `work_submissions` row using service role client:
   ```
   review_scores = scores (as jsonb)
   review_feedback = feedback (or null if passing)
   review_status = 'approved' | 'failed'
   reviewer_id = authenticated user's ID
   reviewed_at = now()
   ```

7. **REVW-08 -- Trigger certificate flow**: If `review_status = 'approved'`, do NOT issue the certificate here (that is Phase 8). Instead, log "Certificate issuance triggered for enrollment {enrollmentId}" and return `{ triggerCertificate: true }` in the response so Phase 8 can wire in the actual certificate creation.

8. **Return** 200 with:
   ```json
   {
     "result": "approved" | "failed",
     "total": number,
     "accuracyGateFailed": boolean,
     "triggerCertificate": boolean,
     "resubmissionExhausted": boolean
   }
   ```

**Security:**
- Reviewer auth verified before any operation.
- Service role client for all DB operations (reviewer needs to update rows belonging to other users).
- Score values validated as integers 1-4 (prevent injection of arbitrary values).
- Return generic 404 for submission not found (do not leak submission IDs).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| reviewer client -> review-submission API | Reviewer submits scores; must verify reviewer identity and score validity |
| review-submission API -> work_submissions | Service role update; reviewer modifies another user's submission record |
| anonymous -> /admin/reviewer/ | Must be blocked for non-reviewers |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-07 | Spoofing | reviewer dashboard | mitigate | verifyReviewer() checks authenticated email against REVIEWER_EMAILS allowlist; layout.tsx redirects non-reviewers |
| T-07-08 | Tampering | review scores | mitigate | Validate each score is integer 1-4 in API; Accuracy hard gate checked server-side before any status update |
| T-07-09 | Repudiation | review action | mitigate | reviewer_id and reviewed_at recorded on every review; audit trail in work_submissions row |
| T-07-10 | Information Disclosure | submission content | mitigate | Only verified reviewers can access /admin/reviewer/; service role queries scoped to needed columns |
| T-07-11 | Denial of Service | review-submission | accept | Low volume (manual human review); no amplification vector |
| T-07-12 | Elevation of Privilege | status transitions | mitigate | Only 'pending' and 'resubmitted' can be reviewed; server validates current status before update; learner cannot self-approve |
</threat_model>
