<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> submit-work-product API | Untrusted form data + file URL from browser |
| client -> Supabase Storage | Direct file upload via presigned URL |
| submit-work-product -> work_submissions | Service role write bypassing RLS |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Spoofing | submit-work-product | mitigate | Validate auth session via getUser(); verify enrollment.user_id matches authenticated user before any operation |
| T-07-02 | Tampering | file upload URL | mitigate | Validate skillFileUrl starts with expected Supabase Storage bucket path prefix; presigned URLs expire after short TTL |
| T-07-03 | Repudiation | work submission | accept | submitted_at timestamp recorded; enrollment_id links to authenticated user; low-value target for repudiation |
| T-07-04 | Information Disclosure | submit-work-product | mitigate | Return generic 403 for ownership failures; do not reveal whether enrollment exists for other users |
| T-07-05 | Denial of Service | file upload | mitigate | Presigned URL generated only for authenticated enrolled users; Supabase Storage has its own size limits; file type restricted to .md/.txt |
| T-07-06 | Elevation of Privilege | resubmission logic | mitigate | Server-side check that only one resubmission permitted; status transitions validated (only 'failed' can become 'resubmitted') |
</threat_model>
