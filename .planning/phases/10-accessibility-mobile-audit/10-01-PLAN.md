<objective>
Make `aibankinginstitute.com` pass WCAG 2.1 AA across every interactive
surface a learner touches — marketing pages, assessment, education hub,
authenticated course overview, and every AiBI-P module page. The existing
code already uses semantic HTML and Tailwind defaults, so this phase is
primarily about (a) running a repeatable scanner, (b) catching the 10–30
issues that slip through any Tailwind-heavy codebase, and (c) hardening
the mobile viewport story that Phase 05 UAT left untested.

Output: a repeatable audit script the team can run before every release, a
baseline audit report that catalogs findings with fix status, and whatever
source-code fixes the audit surfaces.
</objective>

  <action>
Install `@axe-core/playwright` and `playwright` as devDependencies:
```bash
npm install --save-dev @axe-core/playwright playwright
npx playwright install chromium
```

Add an `audit:a11y` npm script to `package.json`:
```json
"scripts": {
  "audit:a11y": "tsx scripts/a11y-audit.ts"
}
```

Create `scripts/a11y-audit.ts`:
- Accepts `--base` flag (default `http://localhost:3000`) and `--user` /
  `--password` flags for authenticated routes.
- Iterates the route list from `<interfaces>` above.
- For each route × each viewport, loads the page, waits for networkidle,
  runs axe-core via `new AxeBuilder({ page }).analyze()`.
- Filters results to `impact = 'critical' | 'serious'`.
- Writes a markdown table to
  `.planning/phases/10-accessibility-mobile-audit/10-AUDIT-REPORT.md`
  with columns: Route | Viewport | Rule ID | Severity | Nodes | Status
  (Open / Fixed / Suppressed).
- Returns exit code 1 if any critical/serious findings remain unresolved;
  0 if all findings have `Status = Fixed` or `Status = Suppressed`.

Script must be runnable from npm and from CI. Keep it self-contained in a
single TypeScript file under 300 lines.
  </action>
  <verify>
`npm run audit:a11y -- --base=http://localhost:3000 --user=jlgilmore2@gmail.com --password='Test1234!'`
runs end-to-end and writes the report. Exit code 1 on first run (expected — finding baseline).
  </verify>

<task type="auto">
  <name>Task 2: Fix all critical and serious axe-core findings</name>
  <files>
    - src/components/**/*.tsx (as surfaced)
    - src/app/**/*.tsx (as surfaced)
    - src/app/globals.css (focus-ring contrast fixes if needed)
  </files>
  <action>
For each row in the audit report with `Status = Open`:

1. Read the affected component / page at the reported DOM path.
2. Apply the axe-documented remediation (https://dequeuniversity.com/rules).
3. Re-run `npm run audit:a11y` scoped to just that route.
4. If the finding is resolved, mark the row `Fixed` in the report.
5. If the finding is a known Tailwind / react-pdf / next/image edge case
   that cannot be meaningfully fixed, mark it `Suppressed` with a one-line
   rationale in the report and suppress it via `AxeBuilder.disableRules()`
   with an in-code comment.

Common categories this audit will likely flag:
- Missing `aria-label` on icon-only buttons (sidebar toggle, close, etc.)
- Insufficient contrast on secondary text (`text-[color:var(--color-ink)]/50`
  under the 4.5:1 WCAG minimum for small text)
- Missing `<label>` association on complex form controls
- Heading level skips (h1 → h3 with no h2)
- Focus trap gaps in modal dialogs

Commit fixes incrementally — one commit per remediation category, not one
giant commit. Makes review and rollback clean.
  </action>
  <verify>
Final `npm run audit:a11y` returns exit 0. No row in 10-AUDIT-REPORT.md
has `Status = Open`.
  </verify>
</task>

<task type="auto">
  <name>Task 3: Automate the 390px mobile viewport check</name>
  <files>
    - scripts/a11y-audit.ts (extend)
    - .planning/phases/10-accessibility-mobile-audit/10-AUDIT-REPORT.md (extend)

  <action>
Extend the audit script to add two mobile-specific checks per route at
390 × 844:

1. **Horizontal scroll check:** After load, evaluate
   `document.documentElement.scrollWidth > window.innerWidth`. If true,
   record as a `mobile-hscroll` finding.
2. **Tap target check:** Query every `button`, `a`, `input`,
   `[role="button"]`, `[role="link"]`. For each, measure bounding box. If
   either dimension is <44px, record as a `mobile-taptarget` finding with
   the element's path, computed size, and accessible name.

Append a new table section in the audit report for mobile findings with
the same `Open / Fixed / Suppressed` workflow.

Fix typical offenders:
- The `Skip to main content` link usually fails horizontal-scroll because
  of wide default padding — visually hide with `sr-only` until focused.
- Footer social icons often render <44px — wrap in a button with 44px
  min-width/height.
- Breadcrumb ▸ separators inline with text can overflow on 390px — wrap
  the breadcrumb in a horizontal scroll container or break to a second line.
  </action>
  <verify>
After fixes, rerun the audit at 390px — no `mobile-hscroll` open findings,
no `mobile-taptarget` findings for elements the user actually interacts
with (decorative chevrons in `aria-hidden` are excluded).
  </verify>

<task type="auto">
  <name>Task 6: Focus-ring contrast check on terra-on-linen surfaces</name>
  <files>
    - src/app/globals.css
    - .planning/phases/10-accessibility-mobile-audit/10-AUDIT-REPORT.md (focus-ring section)
  </files>
  <action>
WCAG 2.1 AA requires 3:1 contrast between the focus indicator and the
adjacent background. Terra (#b5512e) on linen (#f9f6f0) has strong enough
contrast (~5:1 per the design system), but the default Tailwind focus
ring on terra elements may use a lighter variant that fails.

Audit the `:focus-visible` styles in `src/app/globals.css` and every
component that overrides focus via `focus:ring-*` classes. For each
terra-colored interactive element (buttons, links), confirm:
- The focus ring is at least 2px wide.
- The ring color contrasts against BOTH the element's own background AND
  the page background at ≥3:1.
- The ring is visible on both light (linen) and card (parch) surfaces.

Fix: if any focus ring fails, change the `:focus-visible` style to use
`var(--color-ink)` (near-black) at 2px width with a 1px offset. This is
WCAG-safe on every background color in the palette.
  </action>
  <verify>
Use DevTools' contrast picker on a terra button in focus state. Must read
≥3:1 between the ring and the page background.
  </verify>
