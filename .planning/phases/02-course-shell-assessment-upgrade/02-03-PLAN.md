1. **getEnrollment.ts**: Server-only module.
   - `getEnrollment()`: async function.
     - Import `createServerClient` from `@/lib/supabase/client` and `cookies` from `next/headers`.
     - If `!isSupabaseConfigured()`, return `null` (graceful fallback for local dev without Supabase).
     - Get cookieStore, create server client.
     - Check auth: `const { data: { user } } = await supabase.auth.getUser()`. If no user, return `null`.
     - Query: `supabase.from('course_enrollments').select('id, user_id, completed_modules, current_module, enrolled_at').eq('user_id', user.id).eq('product', 'aibi-p').single()`.
     - Return the enrollment row or `null` if not found.
   - Return type: `Promise<Pick<CourseEnrollment, 'id' | 'user_id' | 'completed_modules' | 'current_module' | 'enrolled_at'> | null>`.

2. **courseProgress.ts**: Pure functions (no imports from Supabase -- just logic).
   - `canAccessModule(moduleNumber: number, completedModules: readonly number[]): boolean`:
     - Module 1 is always accessible if enrolled.
     - Module N is accessible if modules 1 through N-1 are all in completedModules array.
     - Per SHELL-04 and SHELL-05: forward-only enforcement.
   - `getModuleStatus(moduleNumber: number, completedModules: readonly number[], currentModule: number): 'completed' | 'current' | 'locked'`:
     - If moduleNumber is in completedModules: 'completed'
     - If moduleNumber === currentModule: 'current'
     - Otherwise: 'locked'
   - `getPillarStatus(pillar: Pillar, modules: readonly Module[], completedModules: readonly number[]): 'completed' | 'in-progress' | 'locked'`:
     - All pillar modules completed: 'completed'
     - Any pillar module accessible but not all completed: 'in-progress'
     - No pillar module accessible: 'locked'

3. **Update layout.tsx** (created in Plan 01):
   - Call `getEnrollment()` at the top of the layout (it's a Server Component).
   - If enrollment exists: pass `completedModules` and `currentModule` to CourseSidebar and MobileSidebarDrawer.
   - If enrollment is null: still render the layout (overview page should be accessible without enrollment for marketing). Pass empty completedModules and currentModule=1 as defaults.
   - The enrollment data flows to children via the sidebar -- the module page does its own independent check.

4. **Update [module]/page.tsx** (created in Plan 02):
   - At the top of the page component, call `getEnrollment()`.
   - If enrollment is null: `redirect('/courses/aibi-p/purchase')` (SHELL-12).
   - If enrolled but `!canAccessModule(moduleNum, enrollment.completed_modules)`: `redirect('/courses/aibi-p/' + enrollment.current_module)` (SHELL-04).
   - This is SERVER-SIDE enforcement per SHELL-05 -- the redirect happens before any content renders.
   - Import `redirect` from `next/navigation`.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Spoofing | Unauthenticated access to module pages | mitigate | getEnrollment checks auth.getUser(); null user → redirect to purchase |
| T-02-06 | Elevation of Privilege | Accessing locked modules by URL manipulation | mitigate | canAccessModule checks all prior modules completed server-side before rendering (SHELL-05) |
| T-02-07 | Information Disclosure | Course content in page source before redirect | mitigate | redirect() called before any JSX renders; Server Components don't send content to client on redirect |
| T-02-08 | Denial of Service | Repeated enrollment lookups | accept | Single DB query per page load; Supabase connection pooling handles volume; rate limiting deferred |
</threat_model>
