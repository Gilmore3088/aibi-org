  <files>src/app/api/webhooks/stripe/route.ts</files>
  <action>
Create `src/app/api/webhooks/stripe/route.ts`:

**Route configuration:**
- Export `const runtime = 'nodejs'` (webhook needs raw body access)
- Export `const dynamic = 'force-dynamic'` (never cache)

**Signature verification (MANDATORY per CLAUDE.md):**
- Read raw body via `await request.text()`
- Get `stripe-signature` header from request
- Call `stripe.webhooks.constructEvent(body, sig, process.env.STRIPE_WEBHOOK_SECRET!)` inside try/catch
- If constructEvent throws, return 400 with "Webhook signature verification failed" — log the error server-side
- If STRIPE_WEBHOOK_SECRET is not set, return 503

**Event routing:**
- Only process `checkout.session.completed` events — return 200 with `{ received: true }` for all other event types
- Extract `session` from `event.data.object` as `Stripe.Checkout.Session`
- Read metadata: `product`, `mode` (individual/institution), `institution_name`, `quantity`, `user_email`, `discount_applied`

**Individual enrollment provisioning (PAY-04):**
- Use `createServiceRoleClient()` for all DB writes (bypasses RLS — institution_enrollments has no user policies)
- Check for existing enrollment with same `stripe_session_id` to prevent duplicate processing (idempotency)
- If mode is 'individual':
  - Look up user by email in `auth.users` (via service role client `auth.admin.listUsers` with email filter, or query directly)
  - INSERT into `course_enrollments`: `{ user_id, email: session.customer_details?.email ?? user_email, product: 'aibi-p', stripe_session_id: session.id, current_module: 1, completed_modules: '{}', enrolled_at: now() }`
  - If `discount_applied === 'institution_persistent'`, also look up the institution_enrollment_id and set it on the enrollment row

**Institution bundle provisioning (PAY-05):**
- If mode is 'institution':
  - INSERT into `institution_enrollments`: `{ institution_name, seats_purchased: parseInt(quantity), seats_used: 0, stripe_session_id: session.id, discount_locked: true }`
  - The `discount_locked: true` means all future purchases by learners associated with this institution get the ~$63 price (PAY-03, enforced in create-checkout from Plan 01)
  - Do NOT create individual course_enrollments rows yet — the institution admin invite flow (deferred) will handle per-learner activation

**Error handling:**
- Wrap all DB operations in try/catch
- On DB error: log full error server-side, return 500 (Stripe will retry)
- On success: return 200 with `{ received: true }`
- Stripe retries failed webhooks for up to 3 days, so transient failures are recoverable

**Idempotency:**
- Before any INSERT, check if a `course_enrollments` row with the same `stripe_session_id` already exists
- Before any institution_enrollments INSERT, check if a row with the same `stripe_session_id` already exists
- If already processed, return 200 (success) without inserting again

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Stripe -> /api/webhooks/stripe | External service sends signed webhook events |
| /api/webhooks/stripe -> Supabase | Server writes enrollment data using service role |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06 | Spoofing | Forged webhook request | mitigate | `stripe.webhooks.constructEvent` verifies HMAC signature using STRIPE_WEBHOOK_SECRET; reject if verification fails with 400 |
| T-03-07 | Tampering | Modified webhook payload | mitigate | Stripe signature covers the entire raw body; any modification invalidates the signature |
| T-03-08 | Repudiation | Enrollment created without payment | mitigate | Only `checkout.session.completed` events trigger enrollment; stripe_session_id stored for audit trail; idempotency check prevents duplicates |
| T-03-09 | Information Disclosure | Webhook error details | mitigate | Log Stripe/Supabase errors server-side only; return generic status codes to caller |
| T-03-10 | Elevation of Privilege | Attacker triggers enrollment for arbitrary user | mitigate | user_id looked up by email from auth.users via service role — attacker cannot inject arbitrary user_id; enrollment only created for the email that completed Stripe checkout |
| T-03-11 | Denial of Service | Webhook replay | mitigate | Idempotency check on stripe_session_id prevents duplicate rows from replayed events |
</threat_model>
