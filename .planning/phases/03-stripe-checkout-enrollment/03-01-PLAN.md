  <files>src/lib/stripe.ts, src/app/api/create-checkout/route.ts</files>
  <action>
1. Install Stripe: `npm install stripe`

2. Create `src/lib/stripe.ts` — Stripe client singleton:
   - Import `Stripe` from 'stripe'
   - Export a single `stripe` instance using `process.env.STRIPE_SECRET_KEY`
   - Throw a clear error if STRIPE_SECRET_KEY is not set
   - Set apiVersion to '2024-12-18.acacia' (latest stable)
   - Add app info: name 'The AI Banking Institute', url 'https://aibankinginstitute.com'

3. Create `src/app/api/create-checkout/route.ts` — POST handler:
   - Accept JSON body: `{ mode: 'individual' | 'institution', quantity?: number, institution_name?: string, user_email?: string }`
   - Validate: if mode is 'institution', quantity must be >= 5 and institution_name must be non-empty string
   - Validate: if mode is 'individual', quantity is ignored (always 1)

   **Individual flow (PAY-01):**
   - Create Checkout Session with `STRIPE_AIBIP_PRICE_ID`, quantity 1
   - Set `mode: 'payment'`, `success_url` to `{origin}/courses/aibi-p?enrolled=true`, `cancel_url` to `{origin}/courses/aibi-p/purchase`
   - Include `metadata: { product: 'aibi-p', mode: 'individual', user_email }`

   **Institution flow (PAY-02):**
   - Create Checkout Session with `STRIPE_AIBIP_INSTITUTION_PRICE_ID`, quantity from body
   - Set `metadata: { product: 'aibi-p', mode: 'institution', institution_name, quantity: String(quantity) }`

   **Persistent discount check (PAY-03):**
   - Before creating the session, if mode is 'individual' and user_email is provided:
     - Use `createServiceRoleClient()` to query `course_enrollments` joined with `institution_enrollments` to check if the user's email is associated with an institution that has `discount_locked = true`
     - If so, use `STRIPE_AIBIP_INSTITUTION_PRICE_ID` instead of the individual price (the ~$63 price applies automatically)
     - Add `metadata.discount_applied: 'institution_persistent'` so the webhook knows

   - Return `{ url: session.url }` on success (302-style client redirect)
   - Return 400 for validation errors, 500 for Stripe errors with safe error message (never expose Stripe error details to client)

   **Environment variable validation:**
   - Check `STRIPE_SECRET_KEY`, `STRIPE_AIBIP_PRICE_ID` at route invocation
   - Check `STRIPE_AIBIP_INSTITUTION_PRICE_ID` only when institution mode is requested
   - Return 503 with "Payment system not configured" if missing

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> /api/create-checkout | Untrusted input (mode, quantity, email) crosses from browser to server |
| /api/create-checkout -> Stripe | Server sends validated data to Stripe API |
| /api/create-checkout -> Supabase | Server reads institution_enrollments for discount check |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | create-checkout user_email param | mitigate | Validate email format; webhook (Plan 02) uses Stripe session metadata + Stripe customer email, not client-provided email, for enrollment writes |
| T-03-02 | Tampering | create-checkout mode/quantity | mitigate | Server-side validation: mode must be 'individual' or 'institution'; quantity must be integer >= 5 for institution; reject all other values |
| T-03-03 | Information Disclosure | Stripe error details | mitigate | Never return Stripe error messages to client; log server-side, return generic "Payment error" to client |
| T-03-04 | Elevation of Privilege | User manipulates price by sending wrong mode | accept | Price is determined server-side from env var Price IDs, not from client input; Stripe enforces the price on their Checkout page |
| T-03-05 | Denial of Service | Repeated create-checkout calls | accept | Rate limiting deferred per CLAUDE.md decisions log; Stripe has built-in rate limits; add Upstash before launch |
</threat_model>
