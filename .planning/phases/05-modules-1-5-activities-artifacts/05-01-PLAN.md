  <files>src/app/api/courses/submit-activity/route.ts</files>
  <action>
Create POST /api/courses/submit-activity that writes activity responses to the activity_responses table.

Request body shape:
```typescript
{ enrollmentId: string, moduleNumber: number, activityId: string, response: Record<string, string> }
```

Implementation:
1. Parse and validate body — enrollmentId (non-empty string), moduleNumber (int 1-9), activityId (non-empty string matching pattern "N.N"), response (non-empty object).
2. Authenticate user using the same cookies() + createServerClient pattern from save-progress/route.ts.
3. Look up enrollment via service role client — SELECT id, user_id, completed_modules, current_module FROM course_enrollments WHERE id = enrollmentId AND user_id = user.id. Return 403 if not found.
4. Verify moduleNumber equals current_module (forward-only — same pattern as save-progress).
5. Check for existing response — SELECT id FROM activity_responses WHERE enrollment_id = enrollmentId AND activity_id = activityId. If exists, return 409 "Activity already submitted" (CONT-04: cannot re-submit).
6. Validate response fields against minimum character lengths: load the module content file dynamically using getModuleByNumber from @content/courses/aibi-p, find the activity by activityId, iterate its fields — for each field with minLength, check response[field.id].length >= minLength. Return 400 with specific field errors if any fail.
7. INSERT into activity_responses: { enrollment_id: enrollmentId, module_number: moduleNumber, activity_id: activityId, response: response (as jsonb) }.
8. Return 201 { success: true, activityId }.

Security: Follow the same auth + ownership pattern as save-progress/route.ts. Use service role client for the write. Never trust client-side validation alone.

**ActivityForm.tsx** — Client Component ('use client') that replaces ActivityFormShell with a submittable form.

Props interface:
```typescript
interface ActivityFormProps {
  readonly activity: Activity;
  readonly enrollmentId: string;
  readonly moduleNumber: number;
  readonly existingResponse?: Record<string, string> | null;
  readonly onSubmitSuccess?: (activityId: string) => void;
}
```

Implementation:
1. Local state: formValues (Record of field.id to string), errors (Record of field.id to string), submitting (boolean), submitted (boolean initialized from existingResponse !== null).
2. If existingResponse is provided (previously submitted), render all fields as read-only with the saved values and a "Submitted" badge. Do not show the submit button.
3. For each field in activity.fields, render the appropriate input using the same styling as ActivityFormShell but WITHOUT the disabled/opacity-70/cursor-not-allowed classes. Instead: fully interactive, with focus:ring-2 focus:ring-[color:var(--color-terra)] for keyboard accessibility (A11Y-01).
4. Client-side validation on submit: check required fields are non-empty, check minLength constraints. Show inline error messages below the field in text-[color:var(--color-error)] font-mono text-xs. Also show a character count below textarea fields: "{current}/{minLength} characters" in font-mono text-[11px].
5. On submit: POST to /api/courses/submit-activity with { enrollmentId, moduleNumber, activityId: activity.id, response: formValues }. Handle 409 (already submitted), 400 (validation errors), 401/403 (auth), 500.
6. On success: set submitted=true, call onSubmitSuccess callback if provided.
7. If activity.completionTrigger === 'artifact-download' and activity.artifactId exists, after successful submit show a "Download [artifact title]" button that links to /api/courses/artifacts/[artifactId] (Plan 03 will create this route — for now, render the button pointing to /public/artifacts/[artifactId].pdf as a static fallback).
8. All form elements must have proper labels, aria-describedby for hints, aria-invalid for errors. Radio fieldsets use <legend>. This satisfies A11Y-01 (keyboard) and A11Y-02 (not color-only — errors have text messages too).

4. Replace the current static activities section with the new ActivitySection.
5. The "Next Module" link should only be active if all activities for the module have been submitted AND save-progress has been called. Add a client-side state tracker in ActivitySection that counts submitted activities vs total.

|----------|-------------|
| Client form -> submit-activity API | Untrusted form data crosses from browser to server |
| API -> Supabase | Service role client bypasses RLS for writes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Spoofing | submit-activity | mitigate | Verify auth session via Supabase getUser(); verify enrollment.user_id matches authenticated user |
| T-05-02 | Tampering | submit-activity body | mitigate | Server-side validation of moduleNumber, activityId format, field minLength; never trust client validation alone |
| T-05-03 | Repudiation | activity submission | accept | Activity responses are timestamped in DB; no audit log needed at this stage |
| T-05-04 | Information Disclosure | submit-activity errors | mitigate | Return generic "Enrollment not found or access denied" (403) — do not leak whether enrollment exists for other users |
| T-05-05 | Denial of Service | submit-activity | accept | Rate limiting deferred per CLAUDE.md decision (2026-04-15); add before launch |
| T-05-06 | Elevation of Privilege | submit-activity | mitigate | Service role used only after ownership verified; enrollment_id + user_id double-checked before write |
