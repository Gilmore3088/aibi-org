<task type="auto">
  <name>Task 2: Build save-onboarding API route</name>
  <files>
    src/app/api/courses/save-onboarding/route.ts
  </files>
  <action>
Create POST handler at /api/courses/save-onboarding/route.ts:

1. Parse request body: { enrollmentId: string, answers: OnboardingAnswers }
2. Validate answers shape server-side:
   - uses_m365 must be one of 'yes' | 'no' | 'not_sure'
   - personal_ai_subscriptions must be string[] (may be empty for "None")
   - primary_role must be one of the LearnerRole union values
   - Return 400 with { error: string } on invalid input
3. Create a Supabase server client using createServerClient from @supabase/ssr (same cookie pattern as getEnrollment but with the service role key since this is an API route that needs write access).
4. Authenticate: get user via supabase.auth.getUser(). Return 401 if no user.
5. Verify enrollment belongs to user: SELECT id FROM course_enrollments WHERE id = enrollmentId AND user_id = user.id. Return 403 if mismatch.
6. Update: UPDATE course_enrollments SET onboarding_answers = answers WHERE id = enrollmentId.
7. Return 200 with { success: true }.

Use the service role client (SUPABASE_SERVICE_ROLE_KEY) for the write since RLS on course_enrollments requires auth.uid() = user_id, which works for reads but the API route should use service role for writes after verifying ownership manually. This matches the project pattern from Phase 1 decisions.

Import OnboardingAnswers and LearnerRole from @/types/course for validation.
