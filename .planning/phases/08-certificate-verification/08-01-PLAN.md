Create a function `generateCertificateId()` that produces IDs in the format `AIBIP-YYYY-XXXXXX` where YYYY is the current year and XXXXXX is a 6-character alphanumeric string (uppercase letters + digits, no ambiguous chars like 0/O/1/I/L). **Generate-certificate API (src/app/api/courses/generate-certificate/route.ts):**
Create a POST route handler. This is an internal API called by the review-submission route — NOT a public endpoint.

Request body: `{ enrollmentId: string }`

Logic:
1. Validate `isSupabaseConfigured()` — 503 if not
2. Use `createServiceRoleClient()` (this is a system action, not user-initiated)
3. **Re-read the work submission status before issuing (CERT-01 critical check):**
   - Query `work_submissions` WHERE `enrollment_id = enrollmentId` AND `review_status = 'approved'`
   - If no approved submission found, return 409 with error "No approved submission found for this enrollment"
   - This prevents premature issuance even if somehow called without actual approval
4. **Check for existing certificate (idempotency per Success Criteria #4):**
   - Query `certificates` WHERE `enrollment_id = enrollmentId`
   - If certificate already exists, return 200 with `{ certificateId: existing.certificate_id, alreadyExists: true }` — do NOT create a duplicate
5. **Look up holder name from enrollment:**
   - Query `course_enrollments` WHERE `id = enrollmentId`, select `email, user_id`
   - Then query `auth.users` via service client to get user metadata for display name
   - Fallback: use email prefix if no display name set
6. **Generate certificate ID** using `generateCertificateId()` from `src/lib/certificates/generateId.ts`
7. **Insert certificate record** into `certificates` table: `enrollment_id`, `certificate_id`, `holder_name`, `designation: 'AiBI-P'`, `issued_at: new Date().toISOString()`
   - If insert fails with unique constraint violation (race condition), query existing and return it
8. **Generate PDF** using `renderToBuffer()` from `@react-pdf/renderer` with the `CertificateDocument` component
   - `holderName`: from step 5
   - `designation`: "AiBI-P -- Banking AI Practitioner"
   - `issuingInstitution`: "The AI Banking Institute"
   - `issuedDate`: formatted issued_at date (e.g., "April 16, 2026")
   - `certificateId`: from step 6
   - `verificationUrl`: `https://aibankinginstitute.com/verify/${certificateId}`
9. Return the PDF as `application/pdf` response with Content-Disposition header for download

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| reviewer -> generate-certificate API | Internal API triggered by reviewer approval; must re-verify approval status |
| learner -> certificate download | Learner must own the enrollment to download their certificate PDF |
| public -> certificate data | Certificate page is authenticated; verification page (Plan 02) is public |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Spoofing | generate-certificate API | mitigate | Re-read review_status from DB before issuing; do not trust caller's claim of approval |
| T-08-02 | Tampering | certificate record | mitigate | Service role client only; no user UPDATE policy on certificates table (RLS) |
| T-08-03 | Repudiation | certificate issuance | mitigate | Certificate record includes enrollment_id, issued_at; work_submissions has reviewer_id and reviewed_at audit trail |
| T-08-04 | Information Disclosure | certificate download | mitigate | GET download handler verifies requesting user owns the enrollment via cookie auth |
| T-08-05 | Denial of Service | PDF generation | accept | renderToBuffer is CPU-bound but only triggered on approval (rare event); no rate limiting needed |
| T-08-06 | Elevation of Privilege | duplicate certificate | mitigate | UNIQUE constraint on enrollment_id in certificates table; code checks before insert |
</threat_model>
